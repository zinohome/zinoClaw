# 架构治理评价框架详解

## 框架设计原则

1. **可量化** - 每个指标都可测量、可比较
2. **可采集** - 优先自动化采集，减少人工成本
3. **可解释** - 评分逻辑透明，结果可追溯
4. **可行动** - 评价结果直接驱动治理任务

## 六大维度详解

---

## 一、结构质量（30%）

评估代码和架构的结构化程度，反映系统的内在质量。

### 1.1 圈复杂度（Cyclomatic Complexity）

**定义**: 衡量代码逻辑复杂度的指标，表示独立路径的数量。

**阈值**:
- 🟢 健康：< 15
- 🟡 警告：15-30
- 🔴 危险：> 30

**测量方式**: 静态分析工具（SonarQube、CodeQL）

**影响**:
- 复杂度高 → 测试困难、维护成本高、bug 率高

**改进建议**:
- 拆分大函数
- 提取公共逻辑
- 使用策略模式替代复杂条件判断

### 1.2 代码重复率（Code Duplication）

**定义**: 重复代码占总代码的比例。

**阈值**:
- 🟢 健康：< 5%
- 🟡 警告：5-10%
- 🔴 危险：> 10%

**测量方式**: 代码扫描工具（SonarQube、CPD）

**影响**:
- 重复率高 → 维护成本高、一致性差

**改进建议**:
- 提取公共模块
- 使用模板方法模式
- 建立共享库

### 1.3 单类行数（Lines Per Class）

**定义**: 单个类的平均代码行数。

**阈值**:
- 🟢 健康：< 500 行
- 🟡 警告：500-1000 行
- 🔴 危险：> 1000 行

**测量方式**: 静态分析

**影响**:
- 类过大 → 职责不清、难以测试

**改进建议**:
- 按职责拆分
- 使用组合代替继承
- 提取值对象

### 1.4 单方法行数（Lines Per Method）

**定义**: 单个方法的平均代码行数。

**阈值**:
- 🟢 健康：< 50 行
- 🟡 警告：50-100 行
- 🔴 危险：> 100 行

**测量方式**: 静态分析

**影响**:
- 方法过大 → 逻辑复杂、难以理解

**改进建议**:
- 提取子方法
- 单一职责原则
- 使用函数式编程

### 1.5 测试覆盖率（Test Coverage）

**定义**: 被测试覆盖的代码比例。

**阈值**:
- 🟢 健康：> 80%
- 🟡 警告：60-80%
- 🔴 危险：< 60%

**测量方式**: CI 流水线（JaCoCo、Coverage.py）

**影响**:
- 覆盖率低 → 回归风险高、重构困难

**改进建议**:
- 补充单元测试
- 关键逻辑优先
- 纳入 CI 门禁

## 二、依赖关系（25%）

评估系统内外部依赖的健康程度，反映架构的耦合度。

### 2.1 下游依赖数（Downstream Dependencies）

**定义**: 系统直接依赖的其他服务/模块数量。

**阈值**:
- 🟢 健康：< 5
- 🟡 警告：5-10
- 🔴 危险：> 10

**测量方式**: 架构图谱、调用链分析

**影响**:
- 依赖过多 → 稳定性风险、变更影响面大

**改进建议**:
- 依赖合并
- 引入防腐层
- 服务聚合

### 2.2 上游依赖数（Upstream Dependencies）

**定义**: 依赖本系统的其他服务/模块数量。

**阈值**:
- 🟢 健康：< 10
- 🟡 警告：10-20
- 🔴 危险：> 20

**测量方式**: 架构图谱、调用链分析

**影响**:
- 被依赖过多 → 变更受限、升级困难

**改进建议**:
- 版本兼容策略
- 向后兼容设计
- 弃用流程规范化

### 2.3 循环依赖（Circular Dependencies）

**定义**: 模块间形成的循环调用关系。

**阈值**:
- 🟢 健康：0
- 🔴 危险：> 0

**测量方式**: 静态分析、架构图谱

**影响**:
- 循环依赖 → 编译困难、部署顺序问题、测试复杂

**改进建议**:
- 引入中间层
- 依赖倒置
- 事件驱动解耦

### 2.4 跨层调用（Cross-Layer Calls）

**定义**: 违反分层架构的调用（如 DAO 层直接调用 Controller 层）。

**阈值**:
- 🟢 健康：0
- 🔴 危险：> 0

**测量方式**: 架构规范检查、ArchUnit

**影响**:
- 跨层调用 → 架构混乱、维护困难

**改进建议**:
- 严格分层
- 依赖规则门禁
- 架构评审

### 2.5 核心链路耦合（Core Path Coupling）

**定义**: 核心业务链路上的服务耦合程度。

**阈值**:
- 🟢 健康：无强耦合
- 🟡 警告：弱耦合
- 🔴 危险：强耦合

**测量方式**: 链路分析、架构评审

**影响**:
- 强耦合 → 单点故障影响大、扩展困难

**改进建议**:
- 服务拆分
- 异步化改造
- 熔断降级

## 评分计算方法

### 单项指标得分

```
如果指标值在健康范围内：
    得分 = 100
如果指标值在警告范围内：
    得分 = 75 - (超标程度 × 25)
如果指标值在危险范围内：
    得分 = 50 - (超标程度 × 50)

其中：
超标程度 = (实际值 - 警告线) / (危险线 - 警告线)
```

### 维度得分

```
维度得分 = Σ(单项指标得分 × 单项权重)
```

### 系统健康度

```
系统健康度 = Σ(维度得分 × 维度权重)
```
